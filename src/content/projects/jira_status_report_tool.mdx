# HSTG Digital Jira Report Generator

Generates Jira sprint reports for multiple teams, posts one consolidated page to Confluence, and notifies a Microsoft Teams channel. Includes **Developer lookup**: open the report and use the link (or embedded iframe) to look up a developer and see their story points and issues.

[![Jira Sprint Report](https://github.com/Kiruthick7/Jira_Status_Report_Tool/actions/workflows/jira-report.yml/badge.svg?event=workflow_dispatch)](https://github.com/Kiruthick7/Jira_Status_Report_Tool/actions/workflows/jira-report.yml)

## Features

- **Multi-team reports** from `team_config.py` (board ID per team, optional members / excluded sprints)
- **Single Confluence page** вЂ” the same page is **updated** each run; sprint details, spillover, verification violations, Blockers (from Jira). Uses **Confluence REST API v2** for better performance and reliability.
- **Sprint details table** вЂ” Sprint name, Actual Start/End, Completed story points, Issues in Sprint, Issues in backlog, **Issues with no story points**, Total Story Points
- **Spillover** вЂ” Issues not completed in the last closed sprint (table: Issue, Summary, **Assignee**, **Label** e.g. tech-debt)
- **Blockers section** вЂ” **Issues on Backlog** then **Issues on Hold**, each line: *Ticket number - Status - Description* (from Jira; backlog first, then on hold). **Comment summaries** for those issues are generated by the LLM (Groq or Gemini); LLM is required for the report.
- **Jira Verification Violations** вЂ” Doughnut chart by type + **Per-issue details** table (Issue, Status, **Assignee**, Violations). Rules are status-dependent: In Progress в†’ +1 AC only; In Test в†’ +1 AC + PR link; Ready for Production / Resolved в†’ all 7 (PR link, branch cleanup, Concourse, rollback, evidence labels, time logged)
- **Story point breakdown** (Developer lookup) вЂ” Not started, In progress, Completed, Blocked/On hold (board column or status name)
- **Teams notification** with Confluence link (date in IST)
- **Developer lookup** вЂ” Link and optional iframe on Confluence; type name, pick team/sprint, fetch to see story points and issue list
- **Cloud Scheduler** вЂ” Run the report on a schedule (default: daily Mon-Sat 7 AM IST; configurable)

## Prerequisites

- Python 3.11+
- Jira account + API token
- Confluence space (for the report page)
- Microsoft Teams incoming webhook (for the report link)
- Groq or Gemini API key (required for LLM-generated **comment summaries** under Blockers)

### Step 1: Clone and install

```bash
cd /path/to/Jira
pip install -r requirements.txt
```

### Step 2: Configure environment

```bash
cp .env.example .env
```

Edit `.env` and set at least:

| Variable | Required | Description |
|----------|----------|-------------|
| `JIRA_USER_NAME` | Yes | Jira user email |
| `JIRA_TOKEN` | Yes | Jira API token |
| `CONFLUENCE_BASE_URL` | For report | e.g. `https://your-site.atlassian.net/wiki` |
| `CONFLUENCE_USER` | For report | Confluence user email |
| `CONFLUENCE_TOKEN` | For report | Confluence API token |
| `CONFLUENCE_SPACE_KEY` | For report | Space key (e.g. `HGD`) |
| `VELOCITY_DASHBOARD_URL` | Optional | Link under the Velocity heading (replaces inline table) |
| `TEAMS_WEBHOOK` | For report | Teams incoming webhook URL |
| `LLM_PROVIDER` | For report | `groq` or `gemini` (required for Blockers comment summaries) |
| `GROQ_API_KEY` | If groq | Groq API key (required when `LLM_PROVIDER=groq`) |
| `GEMINI_API_KEY` or `GOOGLE_API_KEY` | If gemini | Gemini/Google AI key (required when `LLM_PROVIDER=gemini`) |
| `APP_BASE_URL` | For Person Lookup on Confluence | Public URL of this app (see Step 4 / Deploy) |

### Step 3: Configure teams

Edit `team_config.py`: set `board_id` for each team. Use `members: []` to auto-detect assignees from the sprint.

### Step 4: Run locally

**Option A вЂ“ Generate the report once (no browser)**

```bash
python main.py
```

Runs the full flow (fetch from Jira, build report, post to Confluence, send link to Teams) and exits.

**Option B вЂ“ Run the web app (report + Person Lookup in browser)**

```bash
functions_framework --target=generate_and_send_report --port=8080 --debug
```

Use `--debug` so the app runs in a single process (Flask dev server). That can avoid worker crashes (SIGSEGV) you may see when hitting `/report` with the default multi-worker setup. Keep the terminal running. Then:

| What you want | Open in browser |
|---------------|------------------|
| **Developer lookup** (name, team/sprint, fetch) | http://localhost:8080/ or http://localhost:8080/person |
| **Run the report manually** | http://localhost:8080/report |

For Developer lookup to appear on the Confluence page when deployed, set `APP_BASE_URL` in `.env` (e.g. `http://localhost:8080` locally, or your Cloud Function URL in production).

### Step 5: Run with Docker (optional)

Run the app in a containerized environment using Docker:

**Build the Docker image:**

```bash
docker build -t jira-report-app .
```

**Run the container:**

```bash
docker run -p 8080:8080 --env-file .env jira-report-app
```

The app will be available at:
- **Developer lookup:** http://localhost:8080/ or http://localhost:8080/person
- **Run the report:** http://localhost:8080/report

**Notes:**
- The `--env-file .env` flag loads all environment variables from your `.env` file
- The Dockerfile uses the same `functions-framework` entry point as local/GCP deployment
- Uses Python 3.11 slim image with minimal dependencies
- Port 8080 is exposed by default (configurable via `PORT` environment variable)

**Alternative: Run with docker-compose (create docker-compose.yml first):**

```yaml
version: '3.8'
services:
  jira-report:
    build: .
    ports:
      - "8080:8080"
    env_file:
      - .env
```

Then run:

```bash
docker-compose up
```

### Step 6: Deploy to Google Cloud (production)

1. In `.env`, set:
   - `GCP_PROJECT_ID`, `GCP_REGION`, `CLOUD_FUNCTION_NAME`
   - All vars from Step 2 (Jira, Confluence, Teams, LLM)
2. Run:

```bash
./deploy.sh
```

This will:

- Deploy the app as a **Cloud Function** (HTTP).
- Create a **Cloud Scheduler** job that calls the function **daily Monday through Saturday at 7:00 AM IST** at `/report` (generates the report and posts to Confluence + Teams).

3. **After the first deploy:** copy the function URL from the script output and set in `.env`:

   ```bash
   APP_BASE_URL=https://REGION-PROJECT.cloudfunctions.net/FUNCTION_NAME
   ```

   Then run `./deploy.sh` again so the function gets `APP_BASE_URL`. The Confluence report will then show the **Developer lookup** section (link and optional embedded iframe).

**Deploy to Cloud Run (alternative):**

If you prefer Cloud Run over Cloud Functions:

```bash
# Build and push to Google Container Registry
gcloud builds submit --tag gcr.io/$GCP_PROJECT_ID/jira-report

# Deploy to Cloud Run
gcloud run deploy jira-report \
  --image gcr.io/$GCP_PROJECT_ID/jira-report \
  --platform managed \
  --region $GCP_REGION \
  --allow-unauthenticated \
  --set-env-vars="$(cat .env | grep -v '^#' | xargs | tr ' ' ',')"
```

**Schedule:** Default is **daily Mon-Sat 7:00 AM IST** (`30 1 * * 1-6`). To change it, set in `.env` before deploying:

```bash
SCHEDULE_CRON="30 1 * * 1-6"    # 1=Mon, 6=Sat (0=Sun)
SCHEDULE_TIMEZONE=Asia/Kolkata
```

**Alternative: Terraform**
Use `main.tf` and `terraform.tfvars` (copy from `terraform.tfvars.example`) to provision the Cloud Function and Scheduler. Set `teams_webhook`, Confluence vars, and LLM keys in `terraform.tfvars`. The app uses a single Teams webhook for the report link.

## Other ways to run the report on a schedule (no Cloud Function + Scheduler)

If you prefer not to use GCP Cloud Function + Cloud Scheduler, you can run the report on a schedule with one of these:

| Option | Best for | Notes |
|--------|----------|--------|
| **Cron on a server** | You have a Linux/macOS VM or always-on machine | Run `python main.py` or `curl http://your-app/report` at any schedule you want. No cloud scheduler needed. |
| **GitHub Actions** | Repo is on GitHub; you're okay running in CI | Included workflow runs **daily Mon-Sat at 7 AM IST** (`30 1 * * 1-6` in UTC = 01:30 UTC). Customize the schedule in `.github/workflows/jira-report.yml`. Store secrets in repo secrets. |
| **Cloud Run Job + Scheduler** | You want to stay on GCP but not use a Function | Deploy the app as a **Cloud Run Job** (runs to completion). Use Cloud Scheduler to invoke the job at the same cron. |
| **Other clouds** | You already use AWS/Azure | **AWS:** Lambda + EventBridge schedule. **Azure:** Function App with timer trigger. YouвЂ™d deploy the same Python app and call the report entrypoint on schedule. |

**Cron example (server / VM):**
Use the same cron expression in the serverвЂ™s timezone (e.g. if the server is in IST):

```bash
# crontab -e  (Mon & Thu 9:00 AM)
0 9 * * 1,4 cd /path/to/Jira && /usr/bin/python3 main.py
```

If the server is in UTC, 9 AM IST = 03:30 UTC:

```bash
30 3 * * 1,4 cd /path/to/Jira && /usr/bin/python3 main.py
```

**GitHub Actions:**
A workflow is included at `.github/workflows/jira-report.yml` вЂ” runs **daily Mon-Sat at 7:00 AM IST** (`30 1 * * 1-6` = 01:30 UTC). To customize the schedule, edit the cron expression in the workflow file. Add **repository secrets**: `JIRA_USER_NAME`, `JIRA_TOKEN`, `TEAMS_WEBHOOK`, `CONFLUENCE_*`, `LLM_PROVIDER`, `GROQ_API_KEY` or `GEMINI_API_KEY`. For the **Developer lookup** and **Sprint metrics dashboard** links to appear in the Confluence report, also add `APP_BASE_URL` (your hosted app URL, no trailing slash) and optionally `VELOCITY_DASHBOARD_URL`. See [GITHUB_ACTIONS_AND_DEVELOPER_LOOKUP.md](docs/GITHUB_ACTIONS_AND_DEVELOPER_LOOKUP.md). Trigger the workflow manually from the Actions tab.

## End-to-end flow

1. **Report is generated**
   Either by Cloud Scheduler (daily Mon-Sat 7 AM IST for GCP deployments), GitHub Actions (daily Mon-Sat 7 AM IST), or manually (e.g. `python main.py` or http://localhost:8080/report). The app posts one Confluence page with all teams' sections.

2. **You open the Confluence page**
   From the link in Teams or from your Confluence space. You see the full report: date, teams included, Developer lookup link, then per team вЂ” sprint details, naming alerts (if any), Jira Verification Violations (doughnut + per-issue table), spillover, and Blockers (Issues on Backlog, then Issues on Hold; each line: Ticket number - Status - Description).

3. **Developer lookup**
   Scroll to the bottom of the Confluence page. Use **"Open Developer lookup in a new tab"** (recommended; Confluence often blocks iframes). Type the developer name, select team and sprint, click **Fetch from Jira** to see story points (Total, Not started, In progress, Completed, Blocked/On hold) and the issue list.

## Configuration

- **JQL and issue types:** Only **Story** and **Bug** are included in the report and Developer lookup (see `JIRA_REPORT_ISSUE_TYPES_JQL` in `team_config.py`). `JIRA_CUSTOM_JQL_FILTER` is applied where relevant (e.g. report JQL, spillover fallback). **Issues in Sprint** counts parent issues only (no subtasks).
- **Sprint metrics dashboard:** Set `VELOCITY_DASHBOARD_URL` in `.env` (or in GitHub Secrets when using Actions) to show a **Sprint metrics dashboard** link on the report.
- **Developer lookup / report links:** Set `APP_BASE_URL` (and optionally `VELOCITY_DASHBOARD_URL`) so the Confluence report shows the Developer lookup link and Sprint metrics dashboard link. When using GitHub Actions, add these as **repository secrets** so the scheduled report includes the links.
- **Sprint naming:** Report warns if the name doesnвЂ™t match `PI<X> S<X> Squad_Name`; it does not change behavior.
- **Rate limiting:** 15-second delay between teams when generating the report.
- **Confluence page:** Fixed title **"HSTG Digital Sprint status report"**. Uses **Confluence REST API v2** with automatic space key в†’ space ID conversion. Each run updates that page if it exists; otherwise a new page is created.
- **Jira Verification Violations:** Per team: **doughnut chart** (counts by type) and **Per-issue details** table (Issue, Status, Assignee, Violations). Violation types are defined in `team_config.py` (`VIOLATION_LABELS` and `VIOLATION_SHORT`): Missing +1 AC comment, Missing PR link, No time logged, No dev/local evidence, Branch Cleanup not mentioned, Concourse link not pasted, Rollback strategy not mentioned. PR link is detected in comments (including ADF), issuelinks, or Development/remotelinks.
- **Developer lookup:** Story point breakdowns (Sprint and This person's) are shown as **doughnut charts** alongside the cards. Only Story and Bug issues are shown. Performance logging tracks slow sprint lookup operations (fallback queries with timing).
- **Blockers section:** List is from Jira: **Issues on Backlog** then **Issues on Hold**, each line *Ticket number - Status - Description* (backlog first, then on hold). **Comment summaries** are generated by the LLM (required): set `LLM_PROVIDER` (groq or gemini) and the corresponding API key. For GCP/Vertex AI, **Vertex AI Gemini 2.0 Flash** or **Gemini 2.0 Flash Lite** are options. See [Vertex AI pricing](https://cloud.google.com/vertex-ai/generative-ai/pricing).
- **Logging:** Most operational logs are at DEBUG. Use `logging.getLogger("jira_api.jira_sprint").setLevel(logging.DEBUG)` (or similar) to see spillover/board details. Performance-sensitive operations (e.g., sprint lookup fallback) now log at INFO with timing information.

## Troubleshooting

### "Failed to load developers" on Developer lookup page

The page shows the error in the Developer/Team area. Common causes:

- **Jira credentials:** Ensure `JIRA_USER_NAME` and `JIRA_TOKEN` in `.env` are correct and the user has access to the boards in `team_config.py`.
- **Network:** If the app runs in a restricted environment, Jira API calls may fail; check firewall/proxy.
- **Server crash:** If the server is crashing (see SIGSEGV below), the request to `/api/developers` may never complete. Use `--debug` when running locally.

### SIGSEGV when opening `/report` or when loading developers

The log shows `Worker (pid:...) was sent SIGSEGV!` and the browser may show "This page isn't working" / "ERR_EMPTY_RESPONSE". Try:

1. **Run with a single process (recommended for local):**
  Run with Docker | `docker build -t jira-report-app . && docker run -p 8080:8080 --env-file .env jira-report-app` |
| Deploy to GCP (Functions) | `./deploy.sh` |
| Deploy to GCP (Cloud Run) | `gcloud builds submit --tag gcr.io/$GCP_PROJECT_ID/jira-report && gcloud run deploy jira-report --image gcr.io/$GCP_PROJECT_ID/jira-report --platform managed --region $GCP_REGION --allow-unauthenticated
   functions_framework --target=generate_and_send_report --port=8080 --debug
   ```
   `--debug` uses the Flask dev server instead of multi-worker gunicorn, which often avoids this crash.

2. **Confirm the report runs outside the server:** Run `python main.py` in the same environment. If that completes without a crash, the issue is likely the HTTP worker process; `--debug` should help. If `python main.py` also crashes, the segfault is probably in a native dependency (e.g. Jira client, SSL, or LLM library). Try updating: `pip install --upgrade jira requests`.

3. **Check dependencies:** SIGSEGV can come from C extensions (e.g. in `cryptography`, `requests`). Ensure you're on a supported Python version and that packages are installed for your platform.
